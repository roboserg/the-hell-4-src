I have no personal experience with SoftIce, though it has been known to burn pizzas and cause similar such real world problems. While SoftIce is active, the system's clock is frozen, so people relying on the clock for accuracy have overcooked food. I'd recommend windbg, which you can get for free from Microsoft at <http://www.microsoft.com/whdc/ddk/debugging/default.mspx>. It comes with a standard Windows installer, after which you can get right into debugging. Start up Windbg, start Diablo/Hellfire, switch to windbg. Hit f6 and select the process you want to work on. After the attach is complete, you'll be able to issue commands. For this run, give it the command "ba w4 <seedaddr>", where <seedaddr> is the hexadecimal address of the seed you want to watch. When you're done adding breakpoints (you can add several if there's other spots you want to monitor), enter "g" to resume the process. Play as normal. When the game tries to modify that memory location, the debugger will halt it and you can tab out to examine game state.

Beware that DirectX games can be difficult to debug due to their tendency to corrupt screen resolution when you tab out while frozen. If you have problems with this, I can post a program to help you work around it.

Diablo and Hellfire both make extensive use of fixed-location global variables, and the level seeds are no exception. The game computes its initial seed from a call to _time, then uses that to prepare the dungeon level master seeds. Those are then stored.

In Diablo v1.09, the seed array is written from (among other places) 15 lines up from the one and only reference to "SnetGetGameInfo1" (the string, not a function).

Ah, ok. Either of you are welcome to ask for assistance with windbg, btw. Its documentation is good at explaining what a command does, but not so great if you want to know what command will accomplish some goal.

Just as a quick caution for Zamal (or anyone else who tries this), the address 575230 corresponds to the master seed for level 0 (town). So, the command for monitoring for overwrites of dlvl 7 would be: "ba w4 575230+(7*4)". (Break-on-Access, type Write of 4 bytes at address 575230+7*4, 4 since seeds are 4 bytes wide, 7 since we want the seventh element.) In default (masm syntax) mode, windbg can handle basic arithmetic expressions like that for you. You can also set break-on-read breakpoints, but that's probably not desirable here since Hellfire will read the seed whenever it needs to load the level. Access breakpoints can be 1, 2, or 4 bytes wide. It's also possible to breakpoint execution of a particular function, which could be useful if you need to determine why your assembly isn't behaving correctly for certain enemies.

A couple of nice things about the debugger: you can attach at any time (though detaching kills the process if you aren't running Win2003 -- it took that long for Microsoft to figure out how to do a nonfatal detach, apparently). So, it'd be quite feasible to play along, notice something strange, activate the debugger, and investigate. Also, once the debugger is attached, pressing F12 will immediately break to debugger, which could be good if you need to debug an inflight spell (which could conclude before you can get tabbed out and execute a break normally).

Final caution: breaking to debugger halts all threads, so you'll lag out of any network game you're in (and it might also disrupt Kali's client badly enough that you fall off Kali entirely, I don't know).
